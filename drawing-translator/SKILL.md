## 触发容错规则（重要）
- 该 Skill 的意图为：**图纸 + 翻译 + 标注**
- 触发时应采用**关键词组合匹配**，而不是只依赖固定词序
- 只要同一句话中同时出现以下两组核心动作词之一与"图纸/工程图/PDF图纸"等对象词，即应触发：
  - 动作词组A：`翻译` / `翻中文` / `中文翻译`
  - 动作词组B：`标注` / `批注` / `加标注`
  - 对象词：`图纸` / `工程图` / `施工图` / `PDF图纸` / `drawing`
- 连接词与语气词默认忽略，不应影响触发，例如：`和`、`并`、`并且`、`一起`、`顺便`、`帮我`、`请`、`把`
- 允许词序颠倒，例如：
  - `翻译标注图纸`
  - `标注和翻译图纸`
  - `翻译和标注图纸`
  - `帮我把图纸翻译并标注`
  - `把这个drawing翻中文再标注`
- 若用户已先上传 PDF，后续口语化补一句"这个图纸帮我翻译并标注一下"也应触发
- 若只出现"翻译图纸"或"标注图纸"单一动作，则**不要**自动触发本 Skill，避免误判

## 批次规则（重要）
- 用户可能先上传多个PDF，然后再说"翻译标注图纸"
- 收到指令时，往前回溯会话消息，收集从上一次"翻译标注图纸"指令之后、到本次指令之前用户上传的所有 PDF 附件
- 所有收集到的 PDF 作为本次批次一起处理
- 批次切分依据：**发出指令时即切批次**（不以是否回传成功为准）

### 批量回传规则
- 输出 **1个PDF**：直接发送该 PDF 回飞书
- 输出 **超过1个PDF**：所有输出打包成一个 `.zip`，只发送这个 zip，不逐个发送
- zip 命名规则：`drawings_CN_<YYYYMMDD>.zip`

## ⚠️ 强制规则：PDF类型自动判断（2026-04-16 更新）

**核心原则：矢量PDF优先，扫描件才用视觉识别**

### 判断流程（必须严格执行）
```python
from pdfminer.high_level import extract_text

def is_vector_pdf(pdf_path):
    """判断是否为矢量PDF"""
    text = extract_text(pdf_path, page_numbers=[0])
    text_len = len(text.strip())
    
    # 关键判断标准：第一页文本量 > 100字符 = 矢量PDF
    if text_len > 100:
        return True, text_len
    else:
        return False, text_len
```

### 处理策略（按判断结果）
| PDF类型 | 判断标准 | 处理方式 | 优势 |
|---------|---------|---------|------|
| **矢量PDF** | 提取文本 > 100字符 | `run.py extract` → 直接翻译 | ⚡️ 快速、✅ 准确度100% |
| **扫描件** | 提取文本 ≤ 100字符 | 渲染PNG → `image`工具识别 | 唯一可用方法 |

### 实测对比（Therma-Tru D0499案例）
- **矢量PDF提取**：10540字符，秒级完成，关键信息100%准确
- **视觉识别**：需渲染+AI调用，慢10-20倍，可能遗漏细节

### 强制要求
1. **收到PDF后第一步**：调用 `pdfminer.extract_text()` 判断类型
2. **文本量 > 100字符**：立即走矢量提取路线，禁止视觉识别
3. **文本量 ≤ 100字符**：确认为扫描件，才使用视觉识别

## ⚠️ 关键经验：扫描图纸的正确处理路线（O1234 任务总结）

> 本节仅适用于**已确认为扫描件**的PDF，矢量PDF不走此流程。

### 1. 扫描件特征与处理方法
- 扫描件 PDF 内无可选文本，`run.py extract` 提取结果为空或 < 100字符
- Tesseract OCR 对工程扫描图效果极差（乱码），**禁止使用 Tesseract 降级**
- 正确方法：**用 `image` 工具对每张图进行视觉识别（Vision）**

### 2. 图像识别唯一可靠方法
- 每张图**独立调用 `image` 工具**，绝不批量多张合并送视觉（会串图）
- 每次最多 5 张并行，但每张结果必须独立输出 JSON，验证无串图
- 识别后输出标准 JSON blocks，再调用 `write_annotations.py` 写入 PDF

### 3. 禁止将视觉识别任务委托给子 Agent
- 子 Agent 缺乏与主会话相同的 `image` 工具能力，会自动降级为 Tesseract OCR
- 结果看似"已完成"，实际翻译全部乱码，浪费大量时间
- **正确做法：主会话直接调用 `image` 工具，不转交给子 Agent**

### 4. 渲染 PNG 后再识别（PDF/JPG 均适用）
```python
import fitz
doc = fitz.open(str(src))
page = doc.load_page(0)
mat = fitz.Matrix(120/72, 120/72)  # 120 DPI 足够视觉识别
pix = page.get_pixmap(matrix=mat)
pix.save('/tmp/drawings_rendered/<stem>.png')
doc.close()
```
- 所有图先渲染为 PNG，存入 `/tmp/drawings_rendered/`
- 再用 `image` 工具读 PNG 文件进行识别

### 5. 倒置图纸的处理
- 部分扫描图（如 SR0100-01）被 180° 倒置扫描，`page.rotation` metadata = 0 但内容倒置
- 判断方法：视觉识别时检测标题栏方向，若为倒置则 `upside_down=True`
- 修正方法：先 `page.set_rotation(180)`，再 `page.remove_rotation()`，然后写批注
- `write_annotations.py` 已内置 `remove_rotation()` 逻辑，传入 `upside_down=True` 的页面即可

### 6. JPG 源文件的处理
```python
img_doc = fitz.open(str(jpg_path))
tmp_pdf = Path(tempfile.mktemp(suffix='.pdf'))
tmp_pdf.write_bytes(img_doc.convert_to_pdf())
img_doc.close()
# 再走正常 PDF 流程
```

### 7. 文件名规范化（含特殊字符）
- 文件名含空格（如 `LT 500-01.pdf`）：输出时统一替换为下划线 → `LT_500-01_CN.pdf`
- 文件名含逗号（如 `GG797-03,05.pdf`）：输出时替换为下划线 → `GG797-03_05_CN.pdf`
- 文件名无前导零变体（如 `LT 500-01` 对应 BOM 中 `LT 0500-01`）：匹配时做规范化处理

### 8. 对账方法
```python
# 每个源文件 stem，替换空格和逗号为下划线，找对应 _CN.pdf
stem = src.stem.replace(' ','_').replace(',','_')
expected = out_dir / (stem + '_CN.pdf')
```
最终硬对账：213 源 → 213 输出，零缺失。

## 工作流程（通用，不依赖图纸号匹配模板）

### Step 0: 收集批次PDF
- 回溯最近会话消息（最多50条），收集本批次所有 PDF 附件路径
- 同一批次的 PDF 统一处理，逐个走 Step 1-4

### Step 1: 提取文本（仅适用于含文本的 PDF）
```bash
cd <skill_dir>
python3 run.py extract <input.pdf> -o /tmp/extracted_<basename>.json
```
若提取结果为空或全为乱码，判定为扫描件，转入"视觉识别路线"（见上方关键经验）。

输出 JSON 结构（每页一项）：
```json
[
  {
    "page": 0,
    "width": 1190.6,
    "height": 841.9,
    "title_block": "右下区域提取的文字...",
    "notes": "左下区域提取的文字...",

    "body": "上方主体区域文字...",
    "all_text": "全页文字（兜底用）"
  }
]
```

### Step 2: AI 翻译（由 agent 完成，不在脚本里）
根据提取结果，按下方"翻译规则"生成翻译 JSON：
```json
[
  {
    "page": 0,
    "blocks": [
      {
        "label": "【标题栏】",

      {
        "label": "【技术要求】",
        "lines": ["1. 无毛刺无锐边", "2. 请遵守标注公差", "3. 所有焊缝为6mm角焊缝并连续焊"]

        "label": "【其他文本】",
```

保存为 `/tmp/translations_<basename>.json`


- 若 > 1 个，压缩为 `drawings_CN_<YYYYMMDD>.zip`，发送 zip
- 若只有 1 个，直接发送该 PDF

## 翻译规则（强制）
- 输出结构优先包含五块：`【标题栏】`、`【技术要求】`、`【未注明/通用公差表】`、`【版本变更】`、`【其他文本】`
- 扫描件工程图（中文/GB/HB图纸）：重点翻译 `【标题栏】`、`【技术要求】`、`【其他文本】`，其余视内容决定是否输出
- `【未注明/通用公差表】` 为**新增强制块**：只要图纸中存在 `UNLESS OTHERWISE SPECIFIED`、`GENERAL TOLERANCE`、`STANDARD TOLERANCE` 等内容，就必须单独输出该块
- **必须尽量覆盖图纸上所有非中文文本内容，并统一汇总进左上角标注框，不要求在原位置逐项覆盖**
- 左下角注释区逐行翻译：不改变行数/顺序/项目符号，不做改写或精简
- 标题栏字段必须翻译：图号/零件名称/材料/热处理/表面处理/版本/比例/数量
- 翻译文本默认只输出中文；但以下特例可保留原样：版本号/牌号/标准号/不确定的单位/必要的英文原词（可括注）
- 标准代号（GB/HB/DIN/JIS/ISO 等）保留原文，不翻译

## 翻译质量要求（增强）
- 优先使用工程术语映射进行翻译，而不是生硬直译
- 标题栏字段翻译优先级：图号 > 零件名称 > 材料 > 热处理 > 表面处理 > 版本 > 比例 > 数量
- **公差表必须完整翻译，不允许只摘要**
- 常见材料、标准、表面处理应优先采用采购/机械行业常用译法
- 可保留原标准号、牌号、缩写，并采用"中文说明 + 原标准号"风格
- 目标是：**不遗漏任何有语义的非中文文字内容**；宁可在"其他文本"里补充，也不要省略

## ⚠️ 翻译质量强制检查清单（2026-04-16新增）

**核心原则：完整覆盖，不得简化**

### 错误认知纠正
- ❌ **错误理解**：翻译 = 提炼摘要，提供"要点式翻译"
- ✅ **正确理解**：翻译 = 完整覆盖所有可见文字，提供"全文对照翻译"
- ❌ **错误做法**："纹理方向见正视图"（简化版）
- ✅ **正确做法**："红色虚线表示有纹理的表面区域（DASHED RED LINE INDICATES SURFACES THAT ARE GRAINED），纹理方向见正视图（SEE FRONT VIEW FOR GRAIN DIRECTION）"

### 翻译前必查（规划阶段）
```
[ ] 确认PDF类型（矢量/扫描件）→ 用check_pdf_type.py或pdfminer判断
[ ] 提取完整原文（矢量用run.py extract，扫描用image工具）
[ ] 识别所有区块：标题栏、公差、技术要求、版本变更、其他文本
[ ] 统计每个区块的条目数（如技术要求有8条→翻译必须8条）
```

### 翻译中必查（执行阶段）
```
[ ] 【标题栏】8项必含：图号、完整英文标题、零件名称、材料、比例、版本、设计者+日期、审核者+日期
[ ] 【公差】必须包含标题（TOLERANCE UNLESS OTHERWISE SPECIFIED）+ 小数/分数/角度三类公差
[ ] 【技术要求】逐条完整翻译，不得合并或简化，原文N条=翻译N条
[ ] 【版本变更】每条必含：版本号+日期+描述+提出人+审核人（5要素）
[ ] 【其他文本】必须包含：
    - 所有视图标签（FRONT VIEW 1:5, SECTION A-A 1:1, DETAIL D 4:1等）
    - 所有大写说明文字（DASHED RED LINE, GRAIN DIRECTION FOLLOWS, TYP. WALL等）
    - 完整料号对应表（FRAME COLOR / IG THICKNESS / PART NO.）
    - 完整保密声明（CONFIDENTIAL DOCUMENT + THIS DOCUMENT IS THE PROPERTY OF...）
    - 品牌标识（THERMA-TRU DOORS等）
    - 所有尺寸说明（如 .188 X 2 PER SCREW BOSS, MAX ALLOWABLE .030" ABOVE SURFACE）
```

### 翻译后必查（质检阶段）
```
[ ] 条目数核对：技术要求原文8条？翻译8条✓；版本变更原文3次？翻译3+ORIGINAL说明✓
[ ] 关键词核对：原文所有ALL CAPS短语是否都在翻译中体现（可括注英文原文）？
[ ] 完整性核对：原文的每一句话是否都有对应翻译？不得自行判断"不重要"而省略
[ ] 遗漏检测：搜索原文中的专业术语（CRITICAL DIMENSION, GRAIN DIRECTION, MOLD NUMBERS等），是否都翻译了？
[ ] 视图完整性：数一数原文有几个视图标签（FRONT VIEW, SECTION, DETAIL等），翻译中是否全部列出？
```

### 典型错误案例（血泪教训）

#### 案例1：技术要求遗漏（2026-04-16 Therma-Tru D0499）
- **错误**：只翻译了7条，遗漏第8条"模具编号"
- **原因**：未逐条核对，自行判断"模具编号不重要"
- **正确做法**：原文8条→翻译必须8条，不得擅自减少

#### 案例2：版本变更简化（同上案例）
- **错误**：只写"REV L/M/N 三次更新"，遗漏"ORIGINAL"和"ORIGINAL PLOT IF RED"
- **原因**：认为"ORIGINAL不需要翻译"
- **正确做法**：版本变更区的所有条目都必须翻译，包括ORIGINAL

#### 案例3：其他文本过度简化（同上案例）
- **错误**：只写"纹理方向见正视图"，遗漏"DASHED RED LINE INDICATES..."
- **原因**：自行简化，认为"说清楚意思就行"
- **正确做法**：保留原文完整描述，可括注英文确保准确性

### 质量标准
- **A级（合格）**：所有原文内容都有对应翻译，条目数一致，无遗漏
- **B级（不合格）**：有内容遗漏、条目数不符、自行简化原文
- **C级（严重不合格）**：大段内容未翻译、技术要求条数错误

### 强制原则
1. **宁可冗长，不可遗漏**：标注框空间不够可以分页，但不能省略内容
2. **原文N条=翻译N条**：技术要求、版本变更等编号条款必须一一对应
3. **ALL CAPS必译**：原文中的大写专业术语必须在翻译中体现
4. **视图标签全覆盖**：所有FRONT VIEW、SECTION、DETAIL必须列出
5. **不得自行判断重要性**：所有可见文字都重要，不得擅自省略
6. **多页PDF每页独立完整**：即使内容重复，每页都必须有完整翻译标注（2026-04-16新增）

### 多页PDF处理规则（2026-04-16新增）

**错误认知：**
- ❌ "第2/3/4页和第1页内容相同，只需标注'与第1页相同'即可"
- ❌ "为了省空间，后面几页只写简要说明"

**正确做法：**
- ✅ **每一页都必须有完整的翻译标注**
- ✅ 即使技术要求、公差、版本变更在所有页面相同，每页仍需完整列出
- ✅ 可以在【其他文本】中补充说明"本页技术要求与第1页相同"，但8条内容仍需完整写出

**原因：**
1. 用户可能只打印或查看某一页，不会翻回第1页查资料
2. PDF可能被拆分成单页分发，每页必须独立完整
3. 翻译的目的是让每一页都可以独立理解，不依赖其他页

**典型案例（Therma-Tru D0499）：**
- 4页PDF，每页都有完整的NOTES(8条)、REVISIONS(3条)、TOLERANCE(3类)
- 错误做法：只在第1页写完整内容，第2/3/4页只写"内容同第1页"
- 正确做法：4页都写完整的8条技术要求+3条版本变更+公差说明

**处理流程：**
```python
for page_num in range(total_pages):
    # 每一页都单独翻译，不假设"内容相同"
    page_translation = {
        "page": page_num,
        "blocks": [
            {"label": "【标题栏】", "lines": [完整的标题栏信息...]},
            {"label": "【未注明公差】", "lines": [完整的3类公差...]},
            {"label": "【技术要求】", "lines": [完整的8条要求...]},
            {"label": "【版本变更】", "lines": [完整的版本记录...]},
            {"label": "【其他文本】", "lines": [本页特有的说明...]}
        ]
    }
```

**空间不够怎么办？**
- 如果某一页的标注框确实放不下，可以分成多个标注框
- 或者适当缩小字号（但不低于8pt）
- 但**绝不能**因为空间问题省略内容

## ⚠️ 英制→公制自动换算（2026-05-15 新增，默认开启）

**核心原则：保留原始英制数值，括注公制换算结果，保留四位小数**

### 换算规则
- 在所有标注文本写入 PDF 前，`write_annotations.py` 自动调用 `convert_imperial_to_metric()` 对每一行做换算
- 无需手动处理，任何通过 `write_annotations()` 写入的内容都会自动触发

### 支持格式
| 原文 | 换算后 |
|------|--------|
| `0.25in` | `0.25in（6.3500mm）` |
| `.001in` | `.001in（0.0254mm）` |
| `Ø0.63in` | `Ø0.63in（16.0020mm）` |
| `7/16"` | `7/16"（11.1125mm）` |
| `1 1/8"` | `1 1/8"（28.5750mm）` |
| `31/64"` | `31/64"（12.3031mm）` |
| `2.62"` | `2.62"（66.5480mm）` |

### 重复保护
- 已有 `（Xmm）` 括注的内容不会再次换算，防止重复叠加

### 实现位置
- `write_annotations.py` → `convert_imperial_to_metric(text: str) -> str`
- 在 `write_annotations()` 函数的 blocks → lines 转文本阶段自动调用

## 文件命名
- 默认输出后缀使用 **ASCII**：`_CN.pdf`（避免某些客户端显示乱码）
- 规则：
  - 文件名中的空格 → 下划线
  - 文件名中的逗号 → 下划线
  - 保留原始图号结构，末尾追加 `_CN.pdf`
- 示例：
  - `LT 500-01.pdf` → `LT_500-01_CN.pdf`
  - `GG797-03,05.pdf` → `GG797-03_05_CN.pdf`
  - `000029 - RBS - Main Plate - Rev 3.pdf` → `000029-RBS-MainPlate_3_CN.pdf`

## 标注规范（打印友好）
### 文本样式
- 颜色：深红/酒红，RGB(160,0,0)
- 不使用底色；小标题加粗、正文不加粗
- 行距：1.1~1.2；每块6~12行，超出拆块或精简（注释逐行翻译除外）

### 字号（自动检测纸张大小）
- A3/A4（默认）：标题10pt加粗，正文9pt
- A2/A1/A0：标题12pt加粗，正文10~11pt（优先10pt）

### 批注框（拟合历史样本）
- 位置：左上角安全区
- 默认边距：18pt
- 默认宽度：480pt 以内
- 默认高度：约 312.7pt 以内（接近历史样本）
- 当翻译内容明显增多时，允许脚本自动增高标注框
